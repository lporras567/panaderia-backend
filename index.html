Arquitectura y Estructura del Proyecto

Visión general y organización de carpetas

💡 Fundamento: Arquitectura de Tres Capas y Microservicios Contenerizados

Para el Paso 1, los fundamentos teóricos se centran en la Arquitectura de Software y la Modularización por Responsabilidades. Es esencial que el aprendiz visualice cómo un sistema complejo se divide en componentes especializados que colaboran para ofrecer un servicio robusto y escalable.

El proyecto adopta un diseño desacoplado. El backend actúa como un proveedor de servicios (API REST), la base de datos asegura la persistencia transaccional y el frontend gestiona la experiencia del usuario. Esta separación permite la escalabilidad independiente: podemos actualizar la lógica del servidor sin afectar la interfaz, y viceversa. Todo se orquesta mediante contenedores para asegurar que el entorno de ejecución sea idéntico en desarrollo y producción.

Estructura de carpetas del proyecto
💡 Fundamento: Organización Modular (SoC)

Aplicamos el principio de Separación de Preocupaciones (Separation of Concerns). Cada archivo tiene una única responsabilidad: models.py define qué datos guardamos, schemas.py cómo los validamos y la carpeta routers/ cómo los exponemos al mundo. Esta estructura facilita el mantenimiento y el trabajo colaborativo en equipos de desarrollo.

Configuración del Entorno de Desarrollo

Requisitos, instalación y estructura de dependencias

Arequirements.txt — Dependencias Python
💡 Fundamento: Determinismo del Entorno

Para integrar los fundamentos teóricos en esta sección, debemos enfocarnos en tres pilares: el Determinismo Tecnológico, el Aislamiento de Entornos y la Compatibilidad de Librerías. Como instructor, esto le permite explicar que no se trata solo de instalar programas, sino de asegurar que el software sea portable y robusto. Aquí tienes la propuesta técnica para tu HTML, manteniendo tu estructura de pasos:

Fundamento Teórico sugerido:

1. Gestión de Dependencias (requirements.txt): Es un manifiesto técnico que garantiza que todos los desarrolladores y servidores utilicen versiones idénticas de las librerías. Esto evita el error "en mi máquina funciona" al asegurar el determinismo en el comportamiento del backend.

2. Entornos Virtuales (venv): El aislamiento es crítico en Python. Un entorno virtual crea un espacio independiente del sistema operativo para instalar dependencias, evitando conflictos entre proyectos y manteniendo el sistema "limpio".

3. Versionamiento y Compatibilidad: En el desarrollo real, las librerías evolucionan a ritmos distintos. El caso de passlib y bcrypt es un ejemplo clásico de cómo cambios en la API de una librería pueden afectar a otras, requiriendo el uso de versiones específicas para mantener la operatividad.

El archivo requirements.txt actúa como un contrato de versiones. Garantiza que el software sea reproducible en cualquier máquina, evitando fallos por actualizaciones inesperadas de terceros.

Para el Paso 3, los fundamentos teóricos deben centrarse en el Diseño Relacional, la Integridad Referencial y la Optimización de Tipos de Datos. Este es el momento donde el aprendiz entiende cómo la lógica de negocio de la panadería se convierte en una estructura rígida y segura.

Nota Técnica: El uso de NUMERIC(12,2) en lugar de FLOAT es fundamental en sistemas contables para evitar errores de redondeo binario. Los tipos ENUM garantizan que solo se ingresen estados válidos a nivel de motor de base de datos.

💡 Fundamento: El Patrón Singleton y Sesiones

La función get_db() implementa un generador que gestiona el ciclo de vida de la conexión. Al usar el bloque try...finally, garantizamos que cada petición HTTP abra una sesión y la cierre correctamente, evitando fugas de memoria y saturación del pool de conexiones.

DATABASE_URL = (
    f'postgresql://{settings.POSTGRES_USER}:{settings.POSTGRES_PASSWORD}'
    f'@{settings.POSTGRES_HOST}:{settings.POSTGRES_PORT}/{settings.POSTGRES_DB}'
)

engine       = create_engine(DATABASE_URL, echo=True)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base          = declarative_base()

def get_db():
    '''Generador de sesión de base de datos (dependencia FastAPI)'''
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
Csecurity.py — JWT y Hash bcrypt
💡 Fundamento: Autenticación Stateless y Hashing

Implementamos Hashing unidireccional con Bcrypt para proteger las contraseñas (nunca se guardan en texto plano). Complementamos con JSON Web Tokens (JWT) para una autenticación sin estado, donde el cliente porta su identidad de forma segura mediante un token firmado criptográficamente.

python · backend/app/security.py
from datetime import datetime, timedelta
from typing import Optional
from jose import JWTError, jwt
from passlib.context import CryptContext
from .database import settings

pwd_context = CryptContext(schemes=['bcrypt'], deprecated='auto')

def hash_password(plain_password: str) -> str:
    return pwd_context.hash(plain_password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None) -> str:
    to_encode = data.copy()
    expire = datetime.utcnow() + (
        expires_delta or timedelta(minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES)
    )
    to_encode.update({'exp': expire})
    return jwt.encode(to_encode, settings.SECRET_KEY, algorithm=settings.ALGORITHM)

def decode_access_token(token: str) -> Optional[dict]:
    try:
        return jwt.decode(token, settings.SECRET_KEY, algorithms=[settings.ALGORITHM])
    except JWTError:
        return None
Ddependencies.py — Inyección de dependencias y RBAC
💡 Fundamento: Inyección de Dependencias y Middleware Lógico

FastAPI utiliza la inyección de dependencias para validar la seguridad antes de entrar a un endpoint. Esto permite implementar el Control de Acceso basado en Roles (RBAC), garantizando que solo usuarios con privilegios administrativos ejecuten acciones críticas como crear productos o ver reportes contables.

python · backend/app/dependencies.py
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from sqlalchemy.orm import Session
from . import models
from .database import get_db
from .security import decode_access_token

security_scheme = HTTPBearer()

def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security_scheme),
    db: Session = Depends(get_db)
) -> models.Cliente:
    '''Extrae el usuario del token JWT Bearer'''
    token   = credentials.credentials
    payload = decode_access_token(token)
    if not payload:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail='Token inválido o expirado',
            headers={'WWW-Authenticate': 'Bearer'},
        )
    correo: str = payload.get('sub')
    usuario = db.query(models.Cliente).filter(
        models.Cliente.correo == correo,
        models.Cliente.activo == True
    ).first()
    if not usuario:
        raise HTTPException(status_code=401, detail='Usuario no encontrado')
    return usuario

def require_admin(
    current_user: models.Cliente = Depends(get_current_user)
) -> models.Cliente:
    '''Verifica que el usuario tenga rol admin'''
    if current_user.rol != 'admin':
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail='Acceso denegado: se requiere rol administrador'
        )
    return current_user

ORM para las tablas y validación de datos de entrada/salida

Amodels.py — Modelos ORM
💡 Fundamento: Mapeo Objeto-Relacional (ORM)

Para el Paso 5, los fundamentos teóricos se centran en la Capa de Persistencia y la Validación de Esquemas de Datos. Es fundamental explicar la separación de responsabilidades: mientras SQLAlchemy se encarga de cómo viven los datos en el disco, Pydantic se encarga de cómo entran y salen de la API.

El ORM actúa como un puente que permite manipular la base de datos PostgreSQL utilizando clases de Python. Aquí definimos la estructura física: tipos de datos, llaves primarias, foráneas y relaciones de carga (relationship). Esto abstrae el lenguaje SQL y protege la aplicación mediante el uso de parámetros seguros en las consultas.

python · backend/app/models.py
from sqlalchemy import (
    Column, Integer, String, Numeric, Boolean,
    DateTime, ForeignKey, UniqueConstraint, Enum as SAEnum
)
from sqlalchemy.orm import relationship
from sqlalchemy.sql import func
from .database import Base
import enum

class TipoDocumento(str, enum.Enum):
    CC = 'CC'; NIT = 'NIT'; CE = 'CE'; PASAPORTE = 'PASAPORTE'

class TipoCliente(str, enum.Enum):
    persona = 'persona'; empresa = 'empresa'

class RolUsuario(str, enum.Enum):
    admin = 'admin'; user = 'user'

class Cliente(Base):
    __tablename__ = 'clientes'
    id                = Column(Integer, primary_key=True, index=True)
    tipo_documento    = Column(SAEnum(TipoDocumento), nullable=False)
    numero_documento = Column(String(30), unique=True, nullable=False)
    nombre            = Column(String(200), nullable=False)
    tipo_cliente      = Column(SAEnum(TipoCliente), default=TipoCliente.persona)
    direccion         = Column(String(300), nullable=False)
    municipio         = Column(String(100), nullable=False)
    departamento      = Column(String(100), nullable=False)
    celular           = Column(String(20))
    correo            = Column(String(150), unique=True, nullable=False, index=True)
    password_hash     = Column(String(255), nullable=False)
    rol               = Column(SAEnum(RolUsuario), default=RolUsuario.user)
    activo            = Column(Boolean, default=True)
    fecha_registro    = Column(DateTime(timezone=True), server_default=func.now())
    facturas          = relationship('Factura', back_populates='cliente')

class Producto(Base):
    __tablename__ = 'productos'
    id             = Column(Integer, primary_key=True, index=True)
    codigo         = Column(String(50), unique=True, nullable=False, index=True)
    nombre         = Column(String(200), nullable=False)
    cantidad_stock = Column(Integer, default=0)
    precio         = Column(Numeric(12, 2), nullable=False)
    foto_url       = Column(String(500))
    activo         = Column(Boolean, default=True)
    fecha_creacion = Column(DateTime(timezone=True), server_default=func.now())
    lineas         = relationship('ProductoFactura', back_populates='producto')

class Factura(Base):
    __tablename__ = 'facturas'
    id                = Column(Integer, primary_key=True, index=True)
    codigo_factura    = Column(String(50), unique=True, nullable=False)
    fecha_creacion    = Column(DateTime(timezone=True), server_default=func.now())
    fecha_vencimiento = Column(DateTime(timezone=True), nullable=False)
    nombre_panaderia  = Column(String(200), nullable=False)
    nit_panaderia     = Column(String(50), nullable=False)
    dir_panaderia     = Column(String(300), nullable=False)
    cliente_id        = Column(Integer, ForeignKey('clientes.id'), nullable=False)
    subtotal          = Column(Numeric(14, 2), default=0)
    pct_iva           = Column(Numeric(5, 2), default=19.00)
    monto_iva         = Column(Numeric(14, 2), default=0)
    total             = Column(Numeric(14, 2), default=0)
    es_compra_digital = Column(Boolean, default=False)
    estado            = Column(String(30), default='pendiente')
    cliente           = relationship('Cliente', back_populates='facturas')
    lineas            = relationship('ProductoFactura', back_populates='factura',
                                     cascade='all, delete-orphan')

class ProductoFactura(Base):
    __tablename__  = 'producto_factura'
    __table_args__ = (UniqueConstraint('factura_id', 'producto_id'),)
    id              = Column(Integer, primary_key=True, index=True)
    factura_id      = Column(Integer, ForeignKey('facturas.id', ondelete='CASCADE'), nullable=False)
    producto_id     = Column(Integer, ForeignKey('productos.id'), nullable=False)
    precio_unitario = Column(Numeric(12, 2), nullable=False)
    cantidad        = Column(Integer, nullable=False, default=1)
    total_renglon   = Column(Numeric(14, 2), nullable=False)
    factura         = relationship('Factura', back_populates='lineas')
    producto        = relationship('Producto', back_populates='lineas')
Bschemas.py — Schemas Pydantic (validación)
💡 Fundamento: Contratos de Datos y Validación de Tipos

A diferencia de los modelos ORM, los Schemas de Pydantic definen la estructura lógica de los datos que viajan por la red. Permiten validar que un correo tenga formato válido (EmailStr), imponer reglas de negocio (ej. longitud mínima de contraseña) y filtrar qué información devolvemos al cliente (ej. ocultar el hash de la contraseña en las respuestas).

python · backend/app/schemas.py
from pydantic import BaseModel, EmailStr, field_validator
from typing import Optional, List
from datetime import datetime
from decimal import Decimal
from .models import TipoDocumento, TipoCliente, RolUsuario

# ── CLIENTE ────────────────────────────────────────────
class ClienteBase(BaseModel):
    tipo_documento:   TipoDocumento
    numero_documento: str
    nombre:            str
    tipo_cliente:      TipoCliente = TipoCliente.persona
    direccion:         str
    municipio:         str
    departamento:      str

Este es uno de los puntos más importantes para que tus aprendices de ADSO entiendan cómo se comunica la base de datos (SQLAlchemy) con la validación de datos (Pydantic).

Fundamento Teórico: Mapeo de Objetos a Diccionarios

En el desarrollo con FastAPI, trabajamos con dos mundos diferentes:

El mundo de la Base de Datos (ORM): SQLAlchemy devuelve registros como objetos de Python (instancias de una clase). Para acceder a un dato, usamos la sintaxis de punto: usuario.nombre.

El mundo de la API (JSON): Pydantic, por defecto, espera recibir diccionarios para convertirlos a JSON. Para acceder a un dato en un diccionario, se usaría: usuario["nombre"].

La línea model_config = {'from_attributes': True} es un "traductor" que le dice a Pydantic: "Oye, si te paso un objeto de base de datos en lugar de un diccionario, no falles; simplemente busca los datos en los atributos del objeto".

Este módulo implementa el estándar CRUD (Create, Read, Update, Delete). Destacan dos conceptos avanzados: 1. Soft Delete: En el método delete, no eliminamos el registro físicamente para mantener la integridad histórica de las facturas; en su lugar, cambiamos el estado a activo = False. 2. Multipart/Form-Data: Usamos UploadFile para procesar archivos binarios (fotos), almacenándolos en el sistema de archivos y guardando solo la referencia (URL) en la base de datos.

    hoy = datetime.now().strftime('%Y%m%d')

@router.post('/', response_model=schemas.FacturaResponse,
             status_code=status.HTTP_201_CREATED)
async def crear_factura(factura_in: schemas.FacturaCreate,
                        db: Session = Depends(get_db),
                        current_user=Depends(get_current_user)):
    # Control de acceso: user solo puede crear para sí mismo
    if current_user.rol != 'admin' and current_user.id != factura_in.cliente_id:
        raise HTTPException(status_code=403, detail='No puede crear facturas para otro cliente')

    cliente = db.query(models.Cliente).filter(
        models.Cliente.id == factura_in.cliente_id, models.Cliente.activo == True).first()
    if not cliente:
        raise HTTPException(status_code=404, detail='Cliente no encontrado')

    # Calcular totales y validar stock
    subtotal    = Decimal('0')
    lineas_data = []
    for linea in factura_in.lineas:
        producto = db.query(models.Producto).filter(
            models.Producto.id == linea.producto_id, models.Producto.activo == True).first()
        if not producto:
            raise HTTPException(status_code=404,
                detail=f'Producto id={linea.producto_id} no encontrado')
        if producto.cantidad_stock < linea.cantidad:
            raise HTTPException(status_code=400,
                detail=f'Stock insuficiente para {producto.nombre}. '
                       f'Disponible: {producto.cantidad_stock}')
        total_renglon = Decimal(str(producto.precio)) * linea.cantidad
        subtotal += total_renglon
        lineas_data.append({'producto': producto, 'cantidad': linea.cantidad,
                            'precio_unitario': producto.precio,
                            'total_renglon': total_renglon})

    pct_iva   = factura_in.pct_iva
    monto_iva = subtotal * (pct_iva / 100)
    total     = subtotal + monto_iva

    factura = models.Factura(
        codigo_factura    = generar_codigo(),
        fecha_vencimiento = datetime.utcnow() + timedelta(days=15),
        nombre_panaderia  = settings.PANADERIA_NOMBRE,
        nit_panaderia     = settings.PANADERIA_NIT,
        dir_panaderia     = settings.PANADERIA_DIR,
        cliente_id        = factura_in.cliente_id,
        subtotal          = subtotal, pct_iva=pct_iva,
        monto_iva         = monto_iva, total=total,
        es_compra_digital = factura_in.es_compra_digital,
        estado            = 'pendiente'
    )
    db.add(factura); db.flush()

    for ld in lineas_data:
        db.add(models.ProductoFactura(
            factura_id      = factura.id,
            producto_id     = ld['producto'].id,
            precio_unitario = ld['precio_unitario'],
            cantidad        = ld['cantidad'],
            total_renglon   = ld['total_renglon']
        ))
        ld['producto'].cantidad_stock -= ld['cantidad']

    db.commit(); db.refresh(factura)

    # Notificación WebSocket si es compra digital
    if factura.es_compra_digital:
        await manager.broadcast({
            'tipo': 'nueva_compra_digital',
            'factura_id': factura.id,
            'codigo': factura.codigo_factura,
            'cliente': cliente.nombre,
            'total': float(factura.total),
            'timestamp': datetime.utcnow().isoformat()
        })
    return factura

A diferencia del protocolo HTTP tradicional (Petición-Respuesta), WebSockets permite una comunicación bidireccional y persistente. El ConnectionManager implementa el patrón Observer: mantiene una lista de clientes activos y utiliza la técnica de Broadcasting para enviar mensajes instantáneos sin que el cliente los solicite, ideal para alertas de nuevas ventas en el panel administrativo.

python · backend/app/notifications.py
from fastapi import WebSocket
from typing import List
import json

class ConnectionManager:
    '''Gestiona conexiones WebSocket activas de administradores'''
    def __init__(self):
        self.active_connections: List[WebSocket] = []

    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)

    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)

    async def broadcast(self, message: dict):
        '''Envía mensaje JSON a todas las conexiones activas'''
        data = json.dumps(message, ensure_ascii=False)
        for connection in self.active_connections.copy():
            try:
                await connection.send_text(data)
            except Exception:
                self.disconnect(connection)  # Eliminar conexiones muertas

manager = ConnectionManager()

from .database import engine, get_db, settings, Base
from . import models
from .security import hash_password
from .notifications import manager
from .routers import auth, products, clients, invoices

# Crear tablas al iniciar (en producción usar Alembic)
Base.metadata.create_all(bind=engine)

app = FastAPI(
    title       = 'Panadería Digital API',
    version     = '1.0.0',
    description = 'Sistema de facturación para panadería artesanal'
)

# CORS: permitir peticiones del frontend
app.add_middleware(
    CORSMiddleware,
    allow_origins     = ['*'],  # En producción: ['https://tu-dominio.com']
    allow_credentials = True,
    allow_methods     = ['*'],
    allow_headers     = ['*'],
)

# Archivos estáticos (fotos de productos)
os.makedirs('/app/uploads/products', exist_ok=True)
app.mount('/uploads', StaticFiles(directory='/app/uploads'), name='uploads')

# Registrar routers
app.include_router(auth.router,     prefix='/api')
app.include_router(products.router, prefix='/api')
app.include_router(clients.router,  prefix='/api')
app.include_router(invoices.router, prefix='/api')

# Startup: crear admin por defecto si no existe
@app.on_event('startup')
def crear_admin_por_defecto():
    db: Session = next(get_db())
    try:
        if not db.query(models.Cliente).filter(
            models.Cliente.correo == settings.ADMIN_EMAIL).first():
            db.add(models.Cliente(
                tipo_documento   = models.TipoDocumento.NIT,
                numero_documento = '000000000',
                nombre           = settings.ADMIN_NOMBRE,
                tipo_cliente     = models.TipoCliente.empresa,
                direccion        = 'Sede Principal',
                municipio        = 'Medellín',
                departamento     = 'Antioquia',
                correo           = settings.ADMIN_EMAIL,
                password_hash    = hash_password(settings.ADMIN_PASSWORD),
                rol              = models.RolUsuario.admin
            ))
            db.commit()
            print(f'✅ Admin creado: {settings.ADMIN_EMAIL}')
    finally:
        db.close()

# WebSocket para notificaciones en tiempo real
@app.websocket('/ws/notifications')
async def websocket_notifications(websocket: WebSocket):
    await manager.connect(websocket)
    try:
        while True:
            await websocket.receive_text()  # Mantener conexión viva
    except WebSocketDisconnect:
        manager.disconnect(websocket)

# Health check
@app.get('/api/health')
def health_check():
    return {'status': 'ok', 'app': 'Panadería Digital API v1.0'}

* Control de acceso mixto
user* = el usuario solo puede ver/editar su propio registro. user** = el usuario solo ve sus propias facturas. El admin tiene visibilidad total.
Flujo de Compra Digital (Cliente → Admin)
1. Cliente se registra en /api/auth/register
↓
2. Cliente hace login → recibe JWT token
↓
3. Cliente consulta GET /api/products
↓
4. Cliente crea factura con es_compra_digital=true
↓
5. Backend descuenta stock, guarda factura, envía broadcast WebSocket
↓
6. Admin recibe notificación en tiempo real: "Nueva compra digital"
↓
7. Admin cambia estado: pendiente → procesando → enviado → entregado
↓
8. Cliente consulta GET /api/invoices y ve el estado actualizado
